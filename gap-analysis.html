SECURITY FIX — Mass Assignment Vulnerability in Guest Update Endpoint

## Context (verified against code)
`PUT /bookings/:id/guests/:guestId` (bookings/router.ts line 277-298) passes raw `req.body` directly to Prisma `update()` at line 285: `data: req.body`. This allows an attacker to override ANY field — including `tenantId`, `bookingId`, `cFormSubmitted`, etc.

Note: The `POST /:id/guests` (line 275) extracts fields explicitly (`fullName`, `nationality`, `idProofNumber`, `visaNumber`, `visaExpiryDate`, `purposeOfVisit`, `arrivingFrom`, `goingTo`) — so the POST is NOT directly vulnerable, but it still lacks Zod validation.

## Task
1. **Audit** every route in `packages/api/src/modules/bookings/router.ts` that uses `req.body` directly with Prisma. The confirmed vulnerable line is **285** (`data: req.body`). Search for other instances across all modules: `grep -rn "data: req.body" packages/api/src/modules/`
2. **Create a Zod schema** called `bookingGuestSchema` in `packages/shared/src/validators/index.ts` (where ALL existing Zod schemas live — 14 schemas, 228 lines). The schema must match the actual `BookingGuest` model fields: `fullName`, `nationality`, `idProofNumber`, `visaNumber`, `visaExpiryDate` (optional date string), `purposeOfVisit`, `arrivingFrom`, `goingTo`.
3. **Validate** `req.body` against this schema. The codebase uses BOTH `.parse()` (most validators) and `.safeParse()` (night-audit/router.ts line 21). Pick one pattern and be consistent — `.safeParse()` with explicit error handling is recommended for better UX.
4. **Fix the PUT endpoint** at line 285 to use the validated+extracted fields instead of raw `req.body`.
5. **Apply the same pattern** to every other route that passes `req.body` directly to Prisma. Search: `grep -rn "data: req.body" packages/api/src/modules/`
6. **Write/extend Playwright E2E test** in `04-frontdesk-operations.spec.ts` (3.3KB) that attempts to set `tenantId` or `bookingId` via the guest update endpoint and verifies it's rejected or ignored.

## Acceptance Criteria
- [ ] Zero routes in `packages/api/` pass raw `req.body` to Prisma
- [ ] Zod schema added to `packages/shared/src/validators/index.ts`
- [ ] Schema field names match actual Prisma model (fullName, NOT firstName/lastName)
- [ ] E2E test proves tenant override attempt fails
- [ ] No `any` types introduced (per INSTRUCTIONS.md Domain #2)
- [ ] Confidence: 8/10

BUILD — GST-Compliant PDF Invoice Generator

## Context (from gap analysis)
grep "pdf" across the entire codebase returns 0 results. The `Invoice` model already exists in Prisma and invoices are auto-created on checkout (check-in-out/router.ts line ~229-246). The billing/folios module (billing/router.ts, 227 lines) handles folio charges with proper GST splits. However, there is ZERO PDF generation — invoices are database records only. This is a **legal compliance blocker** for GST-registered hotels in India.

## Task
1. **Research first:** Check what PDF libraries are already in `package.json` — **confirmed: none currently installed**. Options: `@react-pdf/renderer` (React-to-PDF, SSR-friendly), `pdfkit` (Node streams), or `jspdf` (client-side). Choose based on whether generation is server-side or client-side.
2. **Create API endpoint:** `GET /api/invoices/:id/pdf` in `packages/api/src/modules/billing/` that:
   - Fetches the `Invoice` model (schema.prisma line 530-555) with fields: `invoiceNumber`, `guestName`, `guestGstin`, `propertyGstin`, `placeOfSupply`, `subtotal`, `totalCgst`, `totalSgst`, `totalIgst`, `grandTotal`, `pdfUrl`, `isProforma`
   - Joins `FolioCharge` (line 484-508) with: `chargeDate`, `category`, `description`, `sacCode`, `quantity`, `unitPrice`, `totalPrice`, `gstRate`, `cgst`, `sgst`, `igst`, `isAutoGenerated`
   - Fetches tenant `gstNumber`, `name`, `address`, `city`, `state`, `pincode` from `Tenant` model (line 83-154)
   - Generates a GST-compliant PDF with:
     - Property name, GSTIN, address (from tenant settings)
     - Guest name, check-in/check-out dates
     - Itemized charges with HSN/SAC codes
     - CGST/SGST/IGST breakdown (use existing GST logic from billing module)
     - Total in ₹ INR with Indian numbering (₹1,23,456.00)
     - Invoice number, date in IST (Asia/Kolkata). Use `new Date().toLocaleString('en-IN', { timeZone: 'Asia/Kolkata' })` for formatting. If creating a shared utility, place it at `packages/shared/src/utils/date-utils.ts`
     - "Tax Invoice" header per GST Act requirements
   - Returns the PDF as `Content-Type: application/pdf` with proper headers for download/print
3. **Add @media print CSS** to the billing dashboard page so browser print also works cleanly.
4. **Add "Download PDF" and "Print" buttons** to the invoice detail view in the billing dashboard.
5. **Extend Playwright test** `07-billing-folios.spec.ts` (4.1KB, existing) to:
   - Generate an invoice via checkout
   - Hit the PDF endpoint and verify 200 response with `content-type: application/pdf`
   - Verify the PDF byte length is > 0

## Constraints
- MUST use `withTenant()` for the PDF endpoint (tenant data security)
- MUST use `authorize('property_owner', 'general_manager', 'front_desk')` — note: `accountant` role exists in type system but is NOT wired into any existing `authorize()` call; add it only if billing access is intended
- Currency formatting MUST use `.toLocaleString('en-IN')` for Indian numbering (₹1,23,456.00)
- No PDF library is currently installed — check `package.json` first; consider `@react-pdf/renderer`, `pdfkit`, or `jspdf` based on SSR vs client needs
- No hardcoded English strings — use translation keys for labels
- Confidence: 8/10
